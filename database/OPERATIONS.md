# Operations

## 文档目的

这份文档说明对 Chronicle Calendar database 进行查询、编辑、导入、迁移时应遵守的操作约束。

---

## 总规则

1. 原始输入不可轻易删除
2. 正式记录不是唯一真相
3. 检索层可以重建
4. 追溯层应尽量保留
5. 不确定的新增内容先进入 drafts

---

## 查询规则

- 优先理解语义，再决定怎么查
- 不把 database 退化成关键词命中器
- 如果用户只是闲聊，不必查库

---

## 修改规则

- 修改 `records` 时，尽量保留对应的 `trace_logs`
- 如果修改来源文本，应考虑是否追加新的 `raw_inputs`
- 如果只是 AI 推断但用户尚未确认，不应直接落成正式 record

---

## 删除规则

- 正式删除要谨慎
- 如果只是“不再使用”，优先用 `status` 或归档，而不是物理删除
- 与历史解释相关的 trace 不应随意清空

---

## 导入导出规则

- 对外交换优先使用 JSON / ICS / JSONL
- 内部运行优先使用 SQLite
- 导出时应尽量保留 record、raw input、draft、trace 的关联关系

---

## 迁移规则

如果要迁移到另一台电脑或另一套软件：

1. 优先迁移整个 `database/` 目录
2. 确保目标环境先读 `README.md`
3. 再读 `SECRETARY_README.md` 或 `TOOLS.md`
4. 最后再接入新的 UI 或 agent

这样能保证：

- 数据跟着规则一起迁移
- 新接入方不会只拿到一份“不会说话的数据库文件”

---

## 对 agent 的要求

agent 在操作 database 时应做到：

- 先读说明，再操作
- 不随意跳过 draft / trace 机制
- 不把临时推断伪装成正式真相
- 修改后说明自己做了什么
